# Writing Documentation

CamelMind uses MDX — Markdown with embedded React components. Each doc file lives in `content/` and is registered in `nav/nav.yml`.

## Frontmatter

Every page starts with YAML frontmatter:

```yaml
---
title: My Page Title
description: A short summary shown below the title and in search results.
---
```

## Markdown features

CamelMind supports GitHub-flavored Markdown:

- **Bold**, *italic*, and `inline code`
- Tables, task lists, and strikethrough
- Fenced code blocks with syntax highlighting
- Auto-generated heading anchors

## File organization

Organize files however you like under `content/`. The folder structure does not determine URLs — `nav/nav.yml` does:

```yaml
- label: "My Page"
  slug: /any/url/path
  file: content/deeply/nested/my-page.mdx
  roles: []
```

## Collapsible sections

Use the `<Details>` component to create collapsible content blocks. Always use the uppercase `<Details>` tag — the lowercase HTML `<details>` tag does not process markdown inside it.

```mdx
<Details summary="Summary text goes here">
  Any markdown content — **bold**, `code`, lists, etc.
</Details>
```

Add an `id` prop to make the section anchor-linkable. Sharing the URL with a `#hash` will auto-open and scroll to that section:

```mdx
<Details id="my-section" summary="Advanced configuration">
  Content here.
</Details>
```

**Example:**

<Details id="example-details" summary="What counts as a breaking change?">
  A breaking change is any modification that requires existing users to update their workflow:

  - Renamed or removed nav slugs
  - Changed frontmatter fields that other tooling depends on
  - Restructured `nav.yml` keys

  Minor wording edits and new pages are **not** breaking changes.
</Details>

## Images

Place static assets in `public/` and reference them in Markdown:

```markdown
![Architecture diagram](/diagrams/architecture.png
```

Click any image to zoom in-place on the rendered page.
