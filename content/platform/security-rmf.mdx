The Risk Management Framework (RMF) is the DoD's structured process for managing cybersecurity risk for information systems. If your application is pursuing a DoD ATO, you must document your application's security controls within the RMF structure. Game Warden significantly reduces this burden by providing a pre-authorized platform baseline that your application can inherit.

## What Is RMF?

RMF is defined in NIST SP 800-37 and the DoD's implementation guidance (DoDI 8510.01). It consists of seven steps:

1. **Prepare** — Establish context and priorities for managing security risk
2. **Categorize** — Determine the impact level of the system (IL2/IL4/IL5 for DoD)
3. **Select** — Choose the appropriate security controls baseline (NIST SP 800-53)
4. **Implement** — Put the selected controls in place
5. **Assess** — Verify the controls are implemented correctly and operating as intended
6. **Authorize** — The AO makes a risk-based decision to authorize (or deny) the system
7. **Monitor** — Continuously monitor the security posture over time

## Game Warden's RMF Posture

Game Warden holds a platform-level ATO that covers all infrastructure components managed by Second Front. This means the following NIST SP 800-53 control families are substantially or fully addressed by the GW platform ATO:

| Control Family | GW Coverage |
|---|---|
| Access Control (AC) | Substantial — network access, namespace RBAC, Vault secrets access |
| Audit and Accountability (AU) | Full — centralized audit logging, tamper-evident storage |
| Configuration Management (CM) | Substantial — platform configuration managed by GW; vendor must document app configuration |
| Identification and Authentication (IA) | Substantial — GW handles platform auth; vendor must document app-level auth |
| Incident Response (IR) | Partial — GW handles platform incidents; vendor must document app-level IRP |
| Risk Assessment (RA) | Full — platform-level risk assessment maintained by GW |
| System and Communications Protection (SC) | Substantial — encryption, network controls, service mesh |
| System and Information Integrity (SI) | Substantial — vulnerability scanning, patching |

The exact list of inherited vs. vendor-responsible controls is provided in the System Security Plan (SSP) template available in App Central.

## What Controls Vendors Are Still Responsible For

While GW covers the platform layer, vendors must implement and document the following:

- **Application-level authentication and authorization** (who can log in, what they can do)
- **Secure coding practices** (input validation, output encoding, OWASP Top 10 mitigations)
- **Data handling** (how your application processes, stores, and transmits data per its classification)
- **Application-level incident response** (your team's procedures for security events in your app)
- **Configuration management for application code and configuration** (change control, version management)
- **Supply chain management** for application dependencies (SBOM, dependency patching)

## How to Document Inherited Controls in Your BoE

When completing your SSP in App Central:

1. For each control, select **Inherited from Game Warden** if the control is fully covered by the GW platform. No additional documentation is needed for these controls.
2. For controls that are **shared** (GW implements some aspects, you implement others), document your application's contribution to the control.
3. For controls that are **vendor-implemented** (GW provides no coverage), document how your application implements the control.

The SSP template pre-populates the Inherited column based on GW's current ATO documentation. Your burden is to fill in the Vendor-Implemented and Shared columns.

<Callout type="tip">
  The SSP template is the fastest path to BoE completion for most vendors. Download it early, fill in the vendor-implemented sections incrementally, and submit sections for review as you complete them rather than waiting until the entire document is finished.
</Callout>
