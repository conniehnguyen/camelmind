Game Warden is a multi-tenant, Kubernetes-based platform built for secure deployment of containerized applications in DoD and federal environments. This page provides a high-level overview of the platform architecture, key components, and how tenant applications are isolated from each other.

## Architecture Overview

Game Warden is structured as a shared platform with a centrally managed control plane and isolated per-tenant namespaces. Second Front manages all platform-level infrastructure; tenants interact with the platform through the Game Warden portal (App Central) and the GW CLI.

```
┌─────────────────────────────────────────────────────┐
│                   GW Control Plane                   │
│  ArgoCD  │  Harbor  │  Vault  │  Prometheus/Grafana  │
│          │  Registry│  Secrets│  Loki (Logs)         │
└──────────┬──────────────────────────────┬────────────┘
           │                              │

    │  Tenant A   │                │  Tenant B   │
    │  Namespace  │                │  Namespace  │
    │  (IL4)      │                │  (IL4)      │
    └─────────────┘                └─────────────┘
```

## Key Components

### Kubernetes

The underlying compute orchestration layer. Game Warden runs EKS-based clusters (AWS) for Commercial and FedRAMP environments, and GovCloud-based clusters for DoW (IL4/IL5) environments. Kubernetes handles workload scheduling, service discovery, auto-scaling, and health management.

### Harbor Container Registry

Harbor (`registry.gamewarden.io`) is the private container registry for all GW tenants. Before deployment, all container images must be pushed to Harbor. Harbor integrates with the GW pipeline to trigger automatic vulnerability scans (via Trivy) whenever a new image is pushed. Images that pass scanning are cryptographically signed and tagged as deployment-ready.

### CI/CD Pipeline Tooling

Game Warden's pipeline automates the build, scan, sign, and deploy workflow for every application. The pipeline runs on each push to the registry and includes SAST analysis, container vulnerability scanning, image provenance verification, and deployment to the target namespace. See [Pipelines and Tooling](/platform/pipelines) for details.

### Monitoring Stack

Game Warden provides a shared Prometheus + Grafana stack for metrics and alerting, and Loki for log aggregation. Tenants access their application's metrics and logs through pre-built dashboards in Grafana. Custom dashboards and alerting rules can be configured per application.

### Service Mesh (Istio)

All intra-cluster traffic is managed by Istio, which provides mutual TLS (mTLS) between all services, traffic management (retries, circuit breaking, canary deployments), and service-level observability. mTLS is enabled by default for all IL4/IL5 tenants and is available but optional for Commercial and FedRAMP tenants.

## Deployment Topology

### Control Plane

The GW control plane is managed entirely by Second Front. It includes ArgoCD (GitOps-based deployment), Vault (secrets management), Harbor, the observability stack, and all cluster-level security tooling. Tenants do not have direct access to control plane components.

### Tenant Namespaces

Each application (or logical application group) is deployed into a dedicated Kubernetes namespace. Namespaces are configured with:

- **Network policies**: Default-deny ingress and egress; explicit allow rules for required traffic
- **Resource quotas**: CPU, memory, and storage limits at the namespace level
- **RBAC**: Kubernetes RBAC scoped to the tenant namespace; tenants cannot access other namespaces
- **Pod security**: OPA/Gatekeeper policies enforcing no-root, no-privilege, no-hostpath requirements

### Ingress

External traffic reaches tenant applications through Game Warden's ingress controller (NGINX-based), which handles TLS termination, host-based routing, and WAF inspection. Custom domains can be configured for tenant applications.
