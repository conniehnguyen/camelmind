Game Warden uses role-based access control (RBAC) to manage what users can see and do within the platform. Understanding the role model helps Org Admins provision access correctly and ensures your team follows a least-privilege security posture.

## Role Types

Game Warden defines four roles at the organization level:

| Role | Who It's For | Access Level |
|---|---|---|
| **Org Admin** | Team leads, account owners | Full access: manage members, create/delete apps, view all data, configure org settings, access billing |
| **Developer** | Engineers, DevOps | Create and manage applications, push images, view and remediate findings, manage BoE, submit support tickets |
| **Auditor** | Compliance officers, assessors, 3PAO reviewers | Read-only access to all data including security findings, BoE documents, audit logs, and scan results |
| **Read-Only** | Stakeholders, program managers | Read-only access to application status and metadata; cannot view security findings or BoE content |

## Role Capabilities

| Action | Org Admin | Developer | Auditor | Read-Only |
|---|---|---|---|---|
| Create/delete applications | Yes | Yes | No | No |
| Push container images | Yes | Yes | No | No |
| View security findings | Yes | Yes | Yes | No |
| Remediate/accept findings | Yes | Yes | No | No |
| View and edit BoE | Yes | Yes | No | No |
| View audit logs | Yes | No | Yes | No |
| Manage organization members | Yes | No | No | No |
| Invite new members | Yes | No | No | No |
| Configure billing | Yes | No | No | No |
| Submit support tickets | Yes | Yes | No | No |
| View application status | Yes | Yes | Yes | Yes |

## How Roles Are Assigned

Only Org Admins can assign roles. Roles can be assigned during the member invitation process or changed at any time from **Organization Settings > Members**.

To change a member's role:
1. Navigate to **Organization Settings > Members**.
2. Click the three-dot menu next to the member's name.
3. Select **Change Role**.
4. Choose the new role and click **Save**.

Role changes take effect immediately.

## Least-Privilege Guidance

Follow these practices to maintain a secure access model:

- **Do not grant Org Admin broadly.** Limit Org Admin to 1–2 individuals who need to manage the organization. Use Developer for engineers who only need to deploy and manage applications.
- **Use Auditor for compliance reviews.** When a 3PAO or compliance officer needs to review your security posture, grant them Auditor access rather than Developer or Org Admin.
- **Review access quarterly.** Remove access for team members who have rotated off the project or left the organization.
- **One account per person.** Never share login credentials. Each team member must have their own account with their own MFA enrollment.
- **Rotate Org Admin keys when personnel change.** If an Org Admin leaves, immediately remove their access and review any API tokens they may have generated.

## Service Accounts and API Access

Game Warden supports API access for CI/CD pipeline integration. Service accounts are created under **Organization Settings > API Keys**. API keys are scoped to a specific role and should be granted the minimum permissions necessary (typically Developer for pipeline automation).

API keys do not expire by default but should be rotated at least annually. Compromised keys should be revoked immediately from **Organization Settings > API Keys > Revoke**.
