Selecting the right base container image is critical for security and compliance on Game Warden. The base image you choose directly affects the vulnerability surface of your container, the compliance artifacts you must produce, and—for IL4/IL5 environments—whether your image is even permitted in the platform.

## Recommended Base Images

| Base Image | Use Case | Notes |
|---|---|---|
| **UBI 8 (ubi8/ubi)** | General-purpose applications, Java, Python, Node.js | Red Hat Universal Base Image. Broadly compatible, RHEL package ecosystem. Good choice for most applications. |
| **UBI 8 Minimal (ubi8/ubi-minimal)** | Lightweight applications with smaller footprint | Smaller than full UBI8; uses `microdnf`. Less package availability. |
| **UBI 9 (ubi9/ubi)** | Applications requiring newer RHEL 9 packages | Newer package versions, improved security defaults. Preferred for new projects. |
| **UBI 9 Minimal (ubi9/ubi-minimal)** | Lightweight applications on RHEL 9 base | Same as UBI 8 Minimal but for RHEL 9. |
| **Alpine Linux (alpine:3.x)** | Small images for simple applications | Very small image size. Uses musl libc—verify library compatibility. Not available in Iron Bank; use for Commercial/FedRAMP only. |
| **Distroless (gcr.io/distroless/...)** | Minimal runtime images with no shell or package manager | Excellent security posture. Requires multi-stage builds. Debugging is more difficult. |

## Iron Bank Images for IL4/IL5

<Callout type="important">
  For IL4 and IL5 (DoW path) deployments, all base images must be sourced from the DoD Iron Bank registry (`registry1.dso.mil`). Iron Bank images are hardened, continuously scanned, and approved for use in DoD environments. Using non-Iron Bank base images in an IL4/IL5 deployment is an automatic finding that will block your ATO.
</Callout>

Iron Bank provides hardened equivalents for common base images:

| Iron Bank Image | Equivalent | Iron Bank Registry Path |
|---|---|---|
| UBI 8 | Red Hat UBI 8 | `registry1.dso.mil/ironbank/redhat/ubi/ubi8` |
| UBI 9 | Red Hat UBI 9 | `registry1.dso.mil/ironbank/redhat/ubi/ubi9` |
| NGINX | NGINX | `registry1.dso.mil/ironbank/opensource/nginx/nginx` |
| Node.js | Node.js LTS | `registry1.dso.mil/ironbank/opensource/nodejs/nodejs18` |
| Python | Python 3.x | `registry1.dso.mil/ironbank/opensource/python/python39` |

Access to the Iron Bank registry requires a free account at [repo1.dso.mil](https://repo1.dso.mil).

## Unsupported Base Images

<Callout type="warning">
  The following base images are not supported on Game Warden and will result in a finding during entrance criteria review:
  - `ubuntu:latest` or any Ubuntu base (on DoW path)
  - `debian:latest` or any Debian base (on DoW path)
  - `centos:*` (CentOS is end-of-life)
  - Any image tagged `:latest` in production (non-reproducible builds)
  - Any image from an unapproved registry (on DoW path)
  - Windows-based container images
</Callout>

## Multi-Stage Builds

Regardless of your base image choice, use multi-stage Docker builds to minimize the size and attack surface of your final image:

```dockerfile
# Build stage
FROM ubi9/ubi AS builder
RUN dnf install -y golang
RUN go build -o /app/server ./cmd/server

# Runtime stage
FROM ubi9/ubi-minimal
COPY --from=builder /app/server /app/server
USER 1001
ENTRYPOINT ["/app/server"]
```

This pattern significantly reduces the number of packages in your final image, which in turn reduces your CVE exposure and speeds up vulnerability scans.
