Game Warden's Kubernetes-based infrastructure supports a wide range of application design patterns. This page describes which patterns are supported, which are not, and recommendations for each supported pattern.

## Supported Patterns

### Microservices

Multiple independently deployable containers communicating over the network. This is the most common pattern on Game Warden and is fully supported.

**Recommendations:**
- Use Kubernetes Services for inter-service discovery. Do not hardcode pod IPs.
- Define resource limits for each microservice independently.
- Use Game Warden's Istio service mesh for mTLS between services. This is required for IL4/IL5 environments.
- Implement health checks for every service, not just the user-facing ones.

### Monolith

A single-container application that handles all functionality. Supported, but with caveats around scaling and resource limits.

**Recommendations:**
- Ensure your monolith can run as a stateless container (state externalized to a managed database or object storage).
- Define appropriate resource requests/limits; monoliths often require more memory than microservices.
- If your monolith has long startup times, configure a generous `initialDelaySeconds` on your liveness probe.

### Batch Jobs

Short-lived containers that run a task to completion. Supported via Kubernetes `Job` resources.

**Recommendations:**
- Design batch jobs to be idempotent—they should produce the same result if run multiple times.
- Configure appropriate TTLs for completed jobs to avoid cluttering the namespace.
- Batch jobs that process large datasets should define appropriate memory limits and be tested for OOM conditions.

### Scheduled Jobs (CronJobs)

Containers that run on a schedule. Supported via Kubernetes `CronJob` resources.

**Recommendations:**
- Set `concurrencyPolicy: Forbid` if your job should not run concurrently with itself.
- Configure a sensible `startingDeadlineSeconds` to handle missed schedules gracefully.
- Ensure scheduled jobs do not require human interaction to complete.

### Stateful Applications with Persistent Volumes

Applications that require persistent storage (e.g., databases, file-processing apps). Supported via Kubernetes `StatefulSet` and `PersistentVolumeClaim` resources.

**Recommendations:**
- Use Game Warden's managed database services (PostgreSQL, Redis) instead of running databases as application containers when possible.
- If you must run your own stateful workload, use `StatefulSet` rather than `Deployment`.

## Unsupported Patterns

The following patterns are **not supported** on Game Warden and will be rejected during entrance criteria review:

| Pattern | Reason |
|---|---|
| **Privileged containers** (`privileged: true`) | Privileged containers can escape the container sandbox and access host resources. Not permitted under any circumstances on GW. |
| **Host networking** (`hostNetwork: true`) | Bypasses Kubernetes network policies and tenant isolation. Not permitted. |
| **Host PID/IPC namespace sharing** | Allows container processes to interact with host processes. Not permitted. |
| **NodePort services** | NodePort services expose ports on every cluster node, conflicting with multi-tenant isolation. Use ClusterIP or LoadBalancer services instead. |
| **DaemonSets** | DaemonSets run pods on every node and are reserved for platform components. Tenant applications may not use DaemonSets. |
| **Arbitrary HostPath mounts** | Mounting host filesystem paths into containers is not permitted. Use PVCs for persistent storage. |

<Callout type="tip">
  If your application currently relies on an unsupported pattern, contact your Second Front representative before beginning onboarding. In many cases, the pattern can be replaced with a supported equivalent without major refactoring.
</Callout>
