Amazon Bedrock is a fully managed service that provides access to foundation models (FMs) from leading AI providers through a unified API. If your Game Warden-deployed application uses AI capabilities—such as text generation, embeddings, summarization, or retrieval-augmented generation (RAG)—you can integrate with Amazon Bedrock while maintaining compliance with your DoD or federal authorization.

## What Is Amazon Bedrock?

Amazon Bedrock offers access to a range of foundation models including Anthropic Claude, Meta Llama, Mistral, Cohere, and Amazon Titan, without requiring you to manage model infrastructure. For Game Warden tenants, Bedrock provides a way to add AI capabilities to your application while keeping data within AWS infrastructure that aligns with your security posture.

## Integration Prerequisites

Before integrating Bedrock with your GW-deployed application:

- **AWS account**: You need an AWS account with Amazon Bedrock enabled. For DoW (IL4/IL5) deployments, this must be an AWS GovCloud account if Bedrock GovCloud is available for your target model; otherwise, contact Second Front to discuss cross-environment connectivity options.
- **IAM role**: Create an IAM role with the `AmazonBedrockFullAccess` managed policy (or a least-privilege custom policy scoped to the specific models and actions your application needs).
- **Network connectivity**: Your GW namespace must be able to reach the Bedrock API endpoint. For DoW environments, this requires either VPC peering or AWS PrivateLink. For Commercial/FedRAMP, outbound HTTPS to `bedrock-runtime.<region>.amazonaws.com` must be allowlisted in your namespace egress policy.
- **Model access**: You must request access to specific foundation models in the Bedrock console before they can be used.

## Step-by-Step Setup

<Steps>
  <Step step="1" title="Define your use case">
    Before integrating Amazon Bedrock on Game Warden, determine how your application will use the service:

    - **Text generation** — chat, summarization, document drafting (Claude, Llama, Mistral)
    - **Embeddings** — semantic search, RAG pipelines (Titan Embeddings, Cohere Embed)
    - **Reranking** — improve retrieval relevance in RAG (Cohere Rerank)

    Your use case determines which foundation models you need access to and whether you require streaming responses (`InvokeModelWithResponseStream`).
  </Step>
  <Step step="2" title="Request Egress Allowlisting">
    Submit a support ticket in App Central requesting an egress allowlist rule for the Bedrock API endpoint: `bedrock-runtime.us-east-1.amazonaws.com` (or your target region). Include your namespace name and the CIDR range of the Bedrock endpoint. Second Front will update your NetworkPolicy within 1 business day.
  </Step>
  <Step step="3" title="Create IAM Credentials for Your Application">
    In your AWS account, create an IAM user or role for your GW application. Generate an access key ID and secret access key (for IAM user) or configure IAM role assumption. Scope the permissions to the specific Bedrock actions and models your application uses:

    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": ["bedrock:InvokeModel", "bedrock:InvokeModelWithResponseStream"],
          "Resource": "arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-3-sonnet-20240229-v1:0"
        }
      ]
    }
    ```
  </Step>
  <Step step="4" title="Store Credentials in Vault">
    Store your AWS credentials in Game Warden's Vault secrets manager. Do not hardcode credentials in your application or Kubernetes manifests.

    Submit a support ticket requesting a Vault secret path provisioned for your namespace (e.g., `secret/your-app/aws-bedrock`). Once provisioned, write your credentials to Vault and configure your application to read them via the Vault Agent sidecar environment variable injection.
  </Step>
  <Step step="5" title="Configure Your Application">
    Use the AWS SDK for your language and configure it to read credentials from environment variables injected by Vault:

    <Tabs>
      <Tab label="Python">

        ```python
        import boto3, os

        client = boto3.client(
            service_name="bedrock-runtime",
            region_name="us-east-1",
            aws_access_key_id=os.environ["AWS_ACCESS_KEY_ID"],
            aws_secret_access_key=os.environ["AWS_SECRET_ACCESS_KEY"],
        )
        ```

      </Tab>
      <Tab label="Node.js">

        ```js
        import { BedrockRuntimeClient } from "@aws-sdk/client-bedrock-runtime";

        const client = new BedrockRuntimeClient({
          region: "us-east-1",
          credentials: {
            accessKeyId: process.env.AWS_ACCESS_KEY_ID,
            secretAccessKey: process.env.AWS_SECRET_ACCESS_KEY,
          },
        });
        ```

      </Tab>
      <Tab label="Java">

        ```java
        BedrockRuntimeClient client = BedrockRuntimeClient.builder()
            .region(Region.US_EAST_1)
            .credentialsProvider(EnvironmentVariableCredentialsProvider.create())
            .build();
        ```

      </Tab>
    </Tabs>
  </Step>
  <Step step="6" title="Test Connectivity">
    Deploy a test pod in your GW namespace and verify that you can successfully call the Bedrock API. If connectivity fails, check your egress NetworkPolicy and confirm the allowlist rule was applied.
  </Step>
</Steps>

## Supported Bedrock Models

The following model families are available through Amazon Bedrock (availability varies by region):

| Provider | Model Family | Use Cases |
|---|---|---|
| Anthropic | Claude 3 (Haiku, Sonnet, Opus) | Text generation, analysis, code, RAG |
| Meta | Llama 3 | Open-weight generation and chat |
| Mistral AI | Mistral and Mixtral | Lightweight generation, summarization |
| Amazon | Titan Text, Titan Embeddings | Generation, embeddings for RAG |
| Cohere | Command, Embed | Generation, embeddings, rerank |

## Security Considerations for AI Workloads on Classified Infrastructure

<Callout type="important">
  For IL4/IL5 DoW deployments, be aware that data sent to Bedrock may leave your GW environment and travel to AWS Bedrock endpoints. Verify whether the Bedrock service is available in GovCloud and whether it meets your data classification requirements before sending CUI or classified data through Bedrock API calls. Consult your program's Authorizing Official if you are uncertain.
</Callout>

Additional considerations:
- **Prompt injection**: Validate and sanitize all user-supplied input before including it in Bedrock prompts to prevent prompt injection attacks
- **Output filtering**: Review AI model outputs for sensitive data before returning them to users or storing them
- **Rate limiting**: Implement application-level rate limiting on Bedrock calls to prevent cost overruns and denial-of-service scenarios
- **Logging**: Log all Bedrock API calls (model, prompt token count, response token count) for audit purposes. Do not log full prompt or completion text if it may contain sensitive data.
